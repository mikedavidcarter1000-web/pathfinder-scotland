## 2026-04-22 Migration tracking reset Session 11b -- baseline production schema; archive prior 94 migrations

- **Files added:** `supabase/migrations/20260422143617_baseline_production_schema.sql` (2320 lines, 123KB single canonical baseline -- 53 tables, 8 enums, 23 functions, 25 triggers, 92 indexes, 97 RLS policies, ~190 constraints); `supabase/migrations/_archived_pre_11b_reset/` directory containing the 94 superseded migration files plus a README explaining provenance; `docs/migration-reset-11b.md` (workflow doc covering what the reset did, why it was necessary, and how to add migrations going forward); CLAUDE.md update under `### Migration file conventions` adding (a) the rule that all schema changes must go through migration files (not raw `execute_sql` for DDL) and (b) a pointer to the reset and the archive directory. `supabase_migrations.schema_migrations` reduced from 83 rows to 1 row. Commit: 2bde287.

- **Memory note "only initial schema tracked remotely" was outdated.** `MEMORY.md` claimed the migration-tracking table held only the original 6 migrations; production actually had 81 (later 83) tracked. The audit output diverged from the premise within minutes -- fortunately the rest of the session's plan still applied because the *kind* of drift (timestamp mismatches, duplicate prefixes, untracked files) was real even if the *scale* was different. **Rule: at session start, when CLAUDE.md / MEMORY.md make a load-bearing claim about external state (migration tracking, table counts, RLS coverage, etc.), verify the claim with one MCP query before designing the work around it. The verification query takes 30 seconds; rebuilding a session plan around a stale claim takes hours.**

- **Drift came from MCP `apply_migration` overwriting timestamps.** Files committed locally as `20260419000004_seed_batch1_career_roles.sql` got tracked in production as version `20260419075435` (the apply timestamp). Future `db push` would treat the version `20260419000004` as missing and try to re-apply the seed -- which would error on already-existing data. Same story for ~30 other files. The other half of the drift came from MCP-applied migrations whose SQL was never committed locally at all (e.g. `create_role_profiles_table` exists in tracking but had no local file). **Rule: when applying a migration via MCP `apply_migration`, the `name` argument must match the local filename's name part exactly so the version-from-prefix logic produces the same value the local file would produce. The Supabase MCP picks the apply timestamp as the version, not the filename prefix -- so file timestamps must already use late-day apply-style prefixes (e.g. `20260419075435`, not `20260419000004`) when known to be applied via MCP. Where this isn't possible, drift is inevitable and a periodic baseline reset is the cure.**

- **Multiple local files with the same timestamp prefix are unrecoverable.** Five different sessions produced `20260425000001_*.sql` files; two produced `20260419000001_*.sql`; etc. The `npx supabase db push` resolver picks files by version-from-prefix-of-filename; identical prefixes mean the second file's INSERT fails on the unique constraint. The duplicates accumulated because Claude Code sessions used `date +%Y%m%d` plus a 6-digit synthetic suffix (`000001`, `000002`...) instead of `date -u +%Y%m%d%H%M%S`. **Rule: always generate migration filenames with `npx supabase migration new <name>` (which uses second-resolution UTC timestamps) rather than constructing the prefix by hand. The CLI guarantees uniqueness against existing files in the directory.**

- **No local pg_dump, no Docker -- programmatic DDL reconstruction via Supabase MCP is the fallback path.** `supabase db dump --linked --schema public` requires Docker on Windows (the CLI tries to spin up a postgres image to host the dump pipeline). PostgreSQL was not installed via chocolatey or other means. The fallback: query `pg_catalog` and `information_schema` directly via `execute_sql` and assemble DDL by hand from the results. Used: a single-row-per-table query with `string_agg` over `pg_attribute` for table DDL; `pg_get_constraintdef`, `pg_get_indexdef`, `pg_get_triggerdef`, `pg_get_functiondef` for the rest; `pg_policy` JOINed via `pg_class` + `pg_namespace` for policies. Total context burn: ~30k tokens of raw catalog data, then assembled into a 123KB migration file. **Rule: when a managed DB host has no local pg_dump path, the catalogue-query fallback is reliable but expensive in tokens; the alternative is a chocolatey install of `postgresql17` for `pg_dump.exe`. For one-off baselines the catalogue path is fine; for repeated dumps install the tool.**

- **Branch verification (Task 30) blocked by Supabase Free plan.** `create_branch` returned `PaymentRequiredException: Branching is supported only on the Pro plan or above`. Fell back to a metadata-count cross-check: count tables, enums, functions, triggers, indexes, policies, FKs, PKs in production via SQL, count the same entities in the baseline file via grep, compare. All counts matched (53/8/23/25/92/97/56/53 = identical). This is a weaker check than a real branch apply -- a CHECK constraint with mismatched expression text would not be caught -- but for a baseline reset where the file was produced from the same catalogue queries, the entity-count match plus inspection-by-construction is acceptable evidence. **Rule: if Supabase branching is unavailable, metadata-count cross-checks plus a manual scan of the baseline file are an acceptable substitute for full apply-and-diff verification on a baseline produced from the same catalogue queries. For migrations that introduce *new* DDL (not a baseline rebuild), branching becomes load-bearing and Pro-plan upgrade is required.**

- **Skipped the intermediate `_reset_dump.sql` file.** Task 26 specified writing the dump to a temporary intermediate file before the baseline migration is constructed in Task 27. In practice the data extracted via Supabase MCP queries can flow directly into the baseline `Write` call without disk-staging -- the intermediate file would just be the same data twice, burning context. The chocolatey install path or pg_dump-based path would naturally produce the intermediate file (since the dump is what `pg_dump` writes to disk), but the catalogue-query path doesn't need it. **Rule: when a multi-step task spec mentions an intermediate artifact whose only consumer is the next step, evaluate whether the intermediate is load-bearing or vestigial; if it's the latter, merge the steps and note the deviation.**

- **CLAUDE.md `BEGIN;`/`COMMIT;` rule overrode task spec's explicit BEGIN/COMMIT request.** Task 27 spec said "BEGIN;" then DDL then "COMMIT;". CLAUDE.md says migrations MUST NOT contain BEGIN/COMMIT (Supabase MCP and CLI wrap their own transaction). Followed CLAUDE.md as the standing convention; the Task 29 step (the only consumer of the baseline) bypasses transaction-wrapping anyway because we just INSERT into the tracking table without re-running the DDL. **Rule: when a task spec contradicts a CLAUDE.md convention, follow CLAUDE.md and document the deviation; conventions exist precisely because per-task specs can drift from project standards.**

- **T20 -- Chromium + WebKit only, no Firefox binary.** Task spec explicitly asked us to skip Firefox. `npx playwright install chromium webkit` downloads the two browsers plus chromium's headless shell (total ~350MB). Config registers exactly two projects; attempting to run `--project=firefox` fails at project-not-found before download is attempted, which is the safer failure mode than downloading-on-demand. **Rule: when a task spec names specific browsers, pass them as explicit args to `playwright install`; never run the bare command which pulls all three.**

- **T22 -- WebKit + Pathfinder CSP kills React dev-mode form handlers.** Pathfinder's production CSP at `next.config.js` sets `script-src 'self' 'unsafe-inline'` with no `'unsafe-eval'`. React 18 dev-mode calls `eval()` for debugging features (stack-trace reconstruction); WebKit's JavaScriptCore rejects it outright and the page console reports "eval() is not supported in this environment ... React requires eval() in development mode". The React app keeps rendering but interactive handlers (like signIn.mutate) silently fail to complete. Playwright's context-level `bypassCSP: true` did not disable this in WebKit during our tests -- we still saw the eval console error after enabling it. **Rule: when a CSP-restricted React app needs to be e2e-tested in WebKit under `next dev`, strip the CSP header from document responses via `page.route('**/*', ...)` filtered on `resourceType() === 'document'`. Do NOT modify `next.config.js` to add dev-only `unsafe-eval` -- the task-framework rule "no production code modifications" applies even when the change is dev-only, and the Playwright-side fix is cleaner.**

- **T22 -- `Locator.fill()` on `<input type="email">` silently no-ops in WebKit against a React controlled input.** Debugging revealed email field remained empty after `.fill(email)` while the password field populated correctly; the handleSubmit handler then aborted on HTML5 `required` validation and never fired the supabase call. Root cause unclear (React state batching + WebKit scheduler) but the fix is deterministic: `await field.click(); await field.fill(''); await field.pressSequentially(value, { delay: 15 }); await expect(field).toHaveValue(value)`. The `delay: 15` spaces keystrokes enough for React's batched setState to commit between them. **Rule: for form flows in Playwright that target React controlled inputs, prefer `pressSequentially` with a small delay plus a `toHaveValue` gate over bare `fill()`. Cost is ~500ms per field; benefit is cross-browser reliability. Chromium's V8 handles both patterns identically.**

- **T22 -- Cross-origin `page.route` mocks need explicit CORS headers.** Supabase's auth endpoints are at `https://<project>.supabase.co/auth/v1/*`. When fulfilling a cross-origin fetch from Playwright, the browser enforces CORS on the returned response just as it would for a real server response. Without `access-control-allow-origin: *` on the fulfilled body, WebKit rejects the response as a CORS violation and supabase-js receives no status code -- the rate-limit test sees neither 429 handling nor any error in the alert. Chromium is marginally more lenient in some cases but will also fail for preflighted requests. **Rule: any `page.route` fulfillment for a cross-origin endpoint needs the four-header CORS block (`origin: *`, `methods: GET, POST, OPTIONS`, `headers: *`, `credentials: true`) plus an OPTIONS branch that returns 204 with the same headers.**

- **T22 -- Glob-based URL matchers on Playwright routes are fragile for URLs with query strings across browsers.** First attempt used `**/auth/v1/token**` which matched on chromium but not webkit (the `?grant_type=password` query-string portion interacts with trailing `**` differently). Function-based matchers `(u: URL) => u.pathname === '/auth/v1/token'` are unambiguous and browser-agnostic. **Rule: for any Supabase / OAuth / similar URL with query parameters, use a function matcher instead of a glob; the cost is 4 extra lines and eliminates one class of cross-browser flake.**

- **T22 -- Next.js's `__next-route-announcer__` has `role="alert"` and conflicts with `getByRole('alert')`.** On any App Router page, Next.js injects a hidden live-region `<div role="alert" aria-live="assertive" id="__next-route-announcer__">` that's empty most of the time. `page.getByRole('alert')` resolves to this element, not the meaningful error banner on the sign-in page. Subsequent `.toContainText()` matches on the wrong element (empty) and times out. **Rule: on Next.js App Router pages, prefer `page.getByText(<error copy>)` for error-alert assertions, or scope via `page.locator('main').getByRole('alert')`. Never rely on an unscoped `getByRole('alert')` reaching the app's alert element.**

- **T22 -- Homepage nav has "Sign in" and "Sign up" links that shadow the auth-card primary CTAs.** Sign-up page has a main-card link "Sign in" to `/auth/sign-in` and the nav also has a link "Sign in" -- `getByRole('link', { name: 'Sign in', exact: true })` resolved to 2 elements. Either scope by `page.locator('main')` or match by URL (`a[href="/auth/sign-up"]`). **Rule: on auth pages, scope link assertions to `page.locator('main')` so nav links don't create strict-mode violations. This applies to every page where the app nav overlaps the page's CTAs.**

- **T23 -- The "Example comparison" banner is dismissed via handleRemove or handleSelect, not via a standalone "Dismiss" button.** The task spec said "Dismiss the example banner by adding a career via the selector" but the example ships with all 3 slots populated (Electrician, Primary Teacher, Nurse), so "adding" is impossible without first removing. ## 2026-04-23 Codify STOP gate principle in CLAUDE.md and prompt templates

- STOP gate discipline: distinguish irreversible operations from routine ones.
  Over-gating adds friction without reducing risk -- a STOP gate on passing
  validation or on a pre-reviewed INSERT delays the session without catching
  anything. The test is: would a wrong outcome here be expensive to reverse,

  mistakes.
